<p><b>Tux3</b> is an open-source <a class="mw-redirect" href="/wiki/Versioning_filesystem" title="Versioning filesystem">versioning filesystem</a> created by Daniel Phillips. He introduced the filesystem as a public replacement for his Tux2 filesystem which had encountered licensing issues due to the filing of several patents.<sup class="reference" id="cite_ref-1"><a href="#cite_note-1">[1]</a></sup> Phillips had previously created the <a class="mw-redirect" href="/wiki/Htree" title="Htree">Htree</a> directory indexing system which eventually became an official feature of <a href="/wiki/Ext3" title="Ext3">ext3</a>.<sup class="reference" id="cite_ref-2"><a href="#cite_note-2">[2]</a></sup> The technical details of Tux3 were first publicized in an email on 23 July 2008.<sup class="reference" id="cite_ref-3"><a href="#cite_note-3">[3]</a></sup>
</p>

<h2><span class="mw-headline" id="Design">Design</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Tux3&amp;action=edit&amp;section=1" title="Edit section: Design">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In broad outline, Tux3 follows a conventional <a href="/wiki/Unix" title="Unix">Unix</a>-style <a href="/wiki/Inode" title="Inode">inode</a>/file/directory design. A Tux3 inode table is a <a href="/wiki/B-tree" title="B-tree">B-tree</a> with versioned attributes at the leaves.  A file is an inode attribute that is a B-tree with versioned <a href="/wiki/Extent_(file_systems)" title="Extent (file systems)">extents</a> at the leaves.  Directory indexes are mapped into directory file blocks as with Htree.  Free space is mapped by a B-tree with extents at the leaves.  It attempts to avoid traditional <a href="/wiki/Journaling_file_system" title="Journaling file system">journaling</a> by introducing a recovery logic which allows it to recover upon remounting.<sup class="reference" id="cite_ref-4"><a href="#cite_note-4">[4]</a></sup>
</p><p>A Tux3 inode is a variable sized object consisting of a list of attributes, each labeled by the version at which the attribute was added, changed or removed. The B-tree index by which inodes are referenced is not versioned.  The B-tree index of a large file is also not versioned, except for the leaves of the index, which contain lists of extents, each labeled by the version at which the referenced file data was added or changed. This style of versioning is essentially the inverse of tree versioning methods used by <a href="/wiki/Write_Anywhere_File_Layout" title="Write Anywhere File Layout">Write Anywhere File Layout</a> (WAFL), <a href="/wiki/ZFS" title="ZFS">ZFS</a> and <a href="/wiki/Btrfs" title="Btrfs">Btrfs</a>, where multiple tree roots of an entire filesystem are created in order to express differences between versions of the filesystem.
</p><p>A claimed advantage of Tux3 style versioned entities is that only a few bytes need to be added to represent single changes to inode attributes or to index new versions of file data blocks, compared to some number of index blocks needing to be allocated and written for tree-versioned filesystems.
</p>
<h2><span class="mw-headline" id="Development_history">Development history</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Tux3&amp;action=edit&amp;section=2" title="Edit section: Development history">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>As of August 2008, Tux3 was only capable of performing basic file I/O on files. By the end of 2008, Tux3 was capable of running within the kernel and able to sustain repeated runs of the file system exerciser fsx-linux, a big accomplishment.<sup class="reference" id="cite_ref-5"><a href="#cite_note-5">[5]</a></sup>
</p><p>On 17 February 2009, Linux booted from a Tux3 root filesystem for the first time.<sup class="reference" id="cite_ref-6"><a href="#cite_note-6">[6]</a></sup>
</p><p>On 18 January 2013, initial <a href="/wiki/Fsck" title="Fsck">fsck</a> support was added.<sup class="reference" id="cite_ref-7"><a href="#cite_note-7">[7]</a></sup> So far it just checks physical <a href="/wiki/Referential_integrity" title="Referential integrity">referential integrity</a>, but the developers are confident that they can make a fully functional <a href="/wiki/E2fsprogs" title="E2fsprogs">e2fsck</a>-quality fsck soon.
</p><p>On 17 May 2014, Daniel Phillips made a plea<sup class="reference" id="cite_ref-8"><a href="#cite_note-8">[8]</a></sup> to the kernel maintainer, to review the code and implement it into the kernel. The goal is to stabilize the code in the kernel and finally get stable.<sup class="reference" id="cite_ref-9"><a href="#cite_note-9">[9]</a></sup>
</p><p>On 9 October 2017, Daniel Phillips sent <a class="external text" href="http://phunq.net/pipermail/tux3/2017-October/002352.html" rel="nofollow">Tux3 update - Shardmap</a> on the mailing list.
</p>
