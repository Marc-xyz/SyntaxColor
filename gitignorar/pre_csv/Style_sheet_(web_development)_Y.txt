<p>A web <b>style sheet</b> is a form of <a class="mw-redirect" href="/wiki/Separation_of_presentation_and_content" title="Separation of presentation and content">separation of presentation and content</a> for <a href="/wiki/Web_design" title="Web design">web design</a> in which the <a class="mw-redirect" href="/wiki/Markup_(computer_programming)" title="Markup (computer programming)">markup</a> (i.e., <a href="/wiki/HTML" title="HTML">HTML</a> or <a href="/wiki/XHTML" title="XHTML">XHTML</a>) of a <a class="mw-redirect" href="/wiki/Webpage" title="Webpage">webpage</a> contains the page's semantic content and structure, but does not define its visual layout (style). Instead, the style is defined in an external style sheet file using a <a href="/wiki/Style_sheet_language" title="Style sheet language">style sheet language</a> such as <a href="/wiki/Cascading_Style_Sheets" title="Cascading Style Sheets">CSS</a> or <a class="mw-redirect" href="/wiki/Extensible_Stylesheet_Language" title="Extensible Stylesheet Language">XSLT</a>. This design approach is identified as a "separation" because it largely supersedes the antecedent methodology in which a page's markup defined both style and structure.
</p><p>The philosophy underlying this methodology is a specific case of <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a>.
</p>

<h2><span class="mw-headline" id="Benefits">Benefits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=1" title="Edit section: Benefits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Separation of style and content has advantages,<sup class="reference" id="cite_ref-why-use-css-mdn_1-0"><a href="#cite_note-why-use-css-mdn-1">[1]</a></sup><sup class="reference" id="cite_ref-purely-branded_2-0"><a href="#cite_note-purely-branded-2">[2]</a></sup> but has only become practical after improvements in popular <a href="/wiki/Web_browser" title="Web browser">web browsers</a>' CSS implementations.
</p>
<h3><span class="mw-headline" id="Speed">Speed</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=2" title="Edit section: Speed">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Overall, users experience of a site utilising style sheets will generally be quicker than sites that don’t use the technology.  ‘Overall’ as the first page will probably load more slowly – because the style sheet AND the content will need to be transferred.  Subsequent pages will load faster because no style information will need to be downloaded – the CSS file will already be in the browser’s cache.
</p>
<h3><span class="mw-headline" id="Maintainability">Maintainability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=3" title="Edit section: Maintainability">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Holding all the presentation styles in one file can reduce the maintenance time and reduces the chance of error, thereby improving presentation consistency. For example, the font color associated with a type of text element may be specified — and therefore easily modified — throughout an entire website simply by changing one short string of characters in a single file. The alternate approach, using styles embedded in each individual page, would require a cumbersome, time consuming, and error-prone edit of every file.
</p>
<h3><span class="mw-headline" id="Accessibility">Accessibility</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=4" title="Edit section: Accessibility">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Sites that use CSS with either XHTML or HTML are easier to tweak so that they appear similar in different browsers (<a href="/wiki/Internet_Explorer" title="Internet Explorer">Internet Explorer</a>, <a class="mw-redirect" href="/wiki/Mozilla_Firefox" title="Mozilla Firefox">Mozilla Firefox</a>, <a href="/wiki/Opera_(web_browser)" title="Opera (web browser)">Opera</a>, <a href="/wiki/Safari_(web_browser)" title="Safari (web browser)">Safari</a>, etc.).
</p><p>Sites using CSS "<a class="mw-redirect" href="/wiki/Fault-tolerant_system" title="Fault-tolerant system">degrade gracefully</a>" in browsers unable to display graphical content, such as <a class="mw-redirect" href="/wiki/Lynx_(browser)" title="Lynx (browser)">Lynx</a>, or those so very old that they cannot use CSS.  Browsers ignore CSS that they do not understand, such as CSS 3 statements. This enables a wide variety of <a href="/wiki/User_agent" title="User agent">user agents</a> to be able to access the content of a site even if they cannot render the style sheet or are not designed with graphical capability in mind. For example, a browser using a <a class="mw-redirect" href="/wiki/Refreshable_Braille_display" title="Refreshable Braille display">refreshable braille display</a> for output could disregard layout information entirely, and the user would still have access to all page content.
</p>
<h3><span class="mw-headline" id="Customization">Customization</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=5" title="Edit section: Customization">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>If a page's layout information is stored externally, a user can decide to disable the layout information entirely, leaving the site's bare content still in a readable form. Site authors may also offer multiple style sheets, which can be used to completely change the appearance of the site without altering any of its content.
</p><p>Most modern web browsers also allow the user to define their own style sheet, which can include rules that override the author's layout rules. This allows users, for example, to bold every hyperlink on every page they visit.
</p>
<h3><span class="mw-headline" id="Consistency">Consistency</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=6" title="Edit section: Consistency">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Because the semantic file contains only the meanings an author intends to convey, the styling of the various elements of the document's content is very consistent. For example, headings, emphasized text, lists and mathematical expressions all receive consistently applied style properties from the external style sheet. Authors need not concern themselves with the style properties at the time of composition. These presentational details can be deferred until the moment of presentation.
</p>
<h3><span class="mw-headline" id="Portability">Portability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=7" title="Edit section: Portability">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The deferment of presentational details until the time of presentation means that a document can be easily re-purposed for an entirely different presentation medium with merely the application of a new style sheet already prepared for the new medium and consistent with elemental or structural vocabulary of the semantic document. A carefully authored document for a web page can easily be printed to a hard-bound volume complete with headers and footers, page numbers and a generated table of contents simply by applying a new style sheet.
</p>
<h2><span class="mw-headline" id="Practical_disadvantages_today">Practical disadvantages today</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=8" title="Edit section: Practical disadvantages today">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Currently specifications (for example, XHTML, XSL, CSS) and software tools implementing these specification are only reaching the early stages of maturity. So there are some practical issues facing authors who seek to embrace this method of separating content and style.
</p>
<h3><span class="mw-headline" id="Narrow_adoption_without_the_parsing_and_generation_tools">Narrow adoption without the parsing and generation tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Style_sheet_(web_development)&amp;action=edit&amp;section=9" title="Edit section: Narrow adoption without the parsing and generation tools">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>While the style specifications are quite mature and still maturing, the software tools have been slow to adapt. Most of the major web development tools still embrace a mixed presentation-content model. So authors and designers looking for GUI based tools for their work find it difficult to follow the semantic web method. In addition to GUI tools, shared repositories for generalized style sheets would probably aid adoption of these methods.
</p>
